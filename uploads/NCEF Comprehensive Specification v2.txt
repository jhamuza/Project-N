Executive Summary
The NCEF (New Communications Equipment Framework) Online Registration System is a comprehensive, centralized, web-based platform designed to manage the entire lifecycle of communications equipment registration in Malaysia. The system replaces legacy manual processes and the previous SIRIM system, providing MCMC with enhanced regulatory oversight, improved service delivery, and AI-powered efficiency.
**Key Objectives:**


Streamline end-to-end registration and monitoring processes


Enhance regulatory oversight through comprehensive data management and audit trails


Improve service delivery through transparent, accessible, and responsive digital platform


Integrate seamlessly with MCMC and external partner systems (SIRIM, RMCD, SSM, MCMC Pay)


Leverage Artificial Intelligence for document validation, risk assessment, and decision support


Ensure security and compliance with Malaysian Personal Data Protection Act (PDPA) 2010
**Project Timeline:** Commenced January 12, 2026; Target Pilot Launch: November 9, 2026

System Architecture


Technology Stack (from SDD v1.4)
| Component | Technology | Purpose |
|-----------|-----------|---------|
| Frontend | React.js / Vue.js | Web portal UI |
| Mobile | React Native / Flutter | Native iOS/Android/Huawei apps |
| Backend API | Node.js / Python FastAPI | RESTful API services |
| Database | PostgreSQL | Primary data store (replacing SIRIM legacy system) |
| File Storage | MinIO | Secure document and file storage |
| AI/ML Engine | Python (TensorFlow/PyTorch) | Document validation, risk scoring |
| Payment Gateway | MCMC Pay API | Payment processing |
| Authentication | Azure Active Directory (AAD) | SSO for internal users |
| Deployment | Docker / Kubernetes | Container orchestration |


Logical Architecture
The system consists of four primary portals:
1. **Internal Portal (MCMC Officers)**
   - System Administrators, OICs, Recommenders, Verifiers, Approvers
   - Task management, application review, analytics, system configuration
2. **External Portal (Applicants)**
   - Suppliers, Principals, Consultants
   - Application submission, status tracking, document management, payment
3. **Public Portal**
   - General public access
   - Equipment search, verification, guidelines, FAQs, AI chatbot
4. **Mobile Applications**
   - Native iOS, Android, Huawei apps
   - Public search, push notifications, on-the-go access

#### 2.3 Scheme C: SDoC (Low Risk) - AI Auto-Acceptance
**Workflow:** Similar to Scheme B, with key difference:
**AI Auto-Acceptance Logic:**


Confidence Score > 90%: **Auto-Accept**
  - RCN issued automatically
  - Certificate generated and sent to applicant
  - Status: Approved (no officer review required)
  - Notification sent to applicant


Confidence Score 70-89%: **Priority Review**
  - Routed to OIC for expedited review
  - Status: Under Review (Priority)


Confidence Score < 70%: **Standard Review**
  - Routed to OIC for full review
  - Status: Under Review
**AI Scoring Criteria (Scheme C):**
| Criteria | Weight | Validation Logic |
|----------|--------|------------------|
| SSM/ROC/ROB Validation | 20% | Real-time database validation confirms legitimate entity |
| Address & Director Match | 10% | Information matches official SSM records |
| Completeness of Technical Specs | 15% | All required technical fields populated |
| Brand & Model Consistency | 10% | Consistent naming across documents, not on blacklist |
| Test Report & Lab Verification | 15% | Report from accredited lab, valid report number |
| Standards Compliance | 10% | Equipment tested against latest MCMC standards |
| Supplier's Past Performance | 10% | Historical compliance record analysis |
| Completeness & Consistency | 10% | All fields correctly filled, no contradictions |
**Threshold Rules:**


90%+: Auto-accept


70-89%: Priority review queue


<70%: Standard review queue


If any 10% criterion missing entirely: Require manual review

Data Models & Form Specifications


Supplier Entity
```
Supplier {
  supplier_id: String (Primary Key, format: SUP-MMYY-XXXXX)
  category: Enum (Company, Individual, Institution, Consultant)
  full_name: String
```

Business Rules & Validation Logic


Registration Validation Rules
| Rule | Condition | Action |
|------|-----------|--------|
| **Email Uniqueness** | Email already exists in system | Reject registration, prompt user to login or use different email |
| **BRN Validation** | Company registration number format invalid | Display error, request correct format |
| **SSM Verification** | BRN not found in SSM database | Flag for manual verification, allow continuation with manual entry |
| **Document Age** | Document issued more than 6 months ago | Reject document, request updated version |
| **Passport Validity** | Passport expired | Reject, request valid passport |
| **Phone Format** | Phone number not in Malaysian format | Display error, show format example |
| **Password Strength** | Password < 12 characters or lacks complexity | Display error, show requirements |


Application Validation Rules
| Rule | Condition | Action |
|------|-----------|--------|
| **RCN Validity** | RCN not found or expired | Display error, prevent application submission |
| **Duplicate IMEI** | IMEI already registered for same RCN | Highlight in validation results, allow user to correct |
| **Payment Completion** | Payment not completed within 14 days | Auto-delete draft, send notification |
| **Iteration Deadline** | Applicant exceeds 60-day iteration period | Auto-cancel application, send notification |
| **Document Completeness** | Required documents missing | Display error, list missing documents |
| **AI Confidence Score** | Score < 70% for Scheme C | Route to manual review queue |
| **Prohibited Equipment** | Offline acceptance not uploaded | Block online workflow progression |


SLA & Deadline Rules
| Process | SLA | Action on Breach |
|---------|-----|-----------------|
| **Draft Retention** | 14 days | Auto-delete if payment not completed |
| **Iteration Period** | 60 days | Auto-cancel application, send notification |
| **Officer Review** | 10 working days (target) | Escalate to supervisor, notify applicant |
| **Renewal Reminder** | 30 days before expiry | Send email + in-app notification |
| **Certificate Validity** | As per Certificate of Conformity | Auto-expire, send renewal reminder 6 months before |

Appendix C: AI Scoring Mechanism (Detailed)
**Scoring Criteria Breakdown:**
1. **SSM/ROC/ROB Validation (20%)**
   - Real-time database lookup
   - Confirmation of legitimate business entity
   - Score: 0% if validation fails, 20% if successful
2. **Address & Director Match (10%)**
   - Compare submitted address with SSM records
   - Match director NRIC with SSM records
   - Score: 0% if no match, 10% if full match
3. **Completeness of Technical Specifications (15%)**
   - Check all required technical fields populated
   - Validate field formats and ranges
   - Score: 15% * (fields_populated / total_required_fields)
4. **Brand & Model Consistency (10%)**
   - Verify brand and model consistency across documents
   - Check against known blacklist
   - Score: 0% if inconsistency/blacklist, 10% if consistent
5. **Test Report & Lab Verification (15%)**
   - Validate test report from accredited lab
   - Verify report number format and validity
   - Check report date within acceptable range
   - Score: 0% if invalid, 15% if valid
6. **Standards Compliance (10%)**
   - Verify equipment tested against latest MCMC standards
   - Check standards version and applicability
   - Score: 0% if non-compliant, 10% if compliant
7. **Supplier's Past Performance (10%)**
   - Analyze historical compliance record
   - Check for previous rejections or issues
   - Score: 10% * (compliant_applications / total_applications)
8. **Completeness & Consistency (10%)**
   - Check all form fields correctly filled
   - Detect contradictions between fields
   - Score: 10% * (consistent_fields / total_fields)
**Threshold Logic:**


**90%+:** Auto-accept (Scheme C only)


**70-89%:** Priority review queue


**<70%:** Standard review queue


**Missing 10% criterion:** Require manual review
